A distributed system is a set of processes which communicate on unreliable network using messages, not sharing memory or global clocks and most importantly coordinate in such a way that it looks like a single system or process.

The interesting and diffuclt task is that even when a process or computer fails or crashes or slowes down the others have to make sure the play does not stop.

A distributed system can also be defined as something where state resides in more than one machine, the only way to communicate is through messages over network, can component can fail at any time independently or may slow down or partial failure.

James Gosling proposal: the root cause of failure of distributed system is assuming the following: network is reliable, latency is 0, bandwidth is infinite.

why it matters: 

scale, fault tolerant, latency, reliabiliy.

when to even use distributed system: when the above 4 outweight the cost in complexity (distributed complexity).

when not to use distributed system: if dataset fits in ram in one box (20-25TB RAM), when QPS is <= few ten thousands, strong inconsistency and small team.

"Distribution is price we pay, not a feature we gain"

Models:

1. The Archipelago (Stale Rumors)

so every information a node recieves over network is historical. we must design system assuming the currect state as probabilistic rumor.

ex: When a driver process receives a heartbeat from an executor indicating it has 2GB of free memory, that is not a fact about the present; it is a fact about the executor at the exact millisecond that packet was constructed.


2. Partial Failure is the Defining Property

on a single machine a failure can be indicated by some stack trace, but in distributed environment its timeout which is one of the dangers of engineering.

for instance: lets say we send a message to kafka broker after 5 seconds it throws timeout exception. now any of the below can be cause of this:

The wire ate it: The request dropped before reaching the NIC (Network Interface Card).

Pre-execution crash: The request reached the OS buffer, but the database process segfaulted before reading it.

Post-execution crash: The database successfully committed the transaction to disk, but the kernel panicked before it could send the ACK back.

The zombie node: The node is perfectly fine, but a massive Garbage Collection (GC) pause has frozen the JVM, so it hasn't processed the request yet.

The return wire ate it: The operation succeeded, but the ACK packet was dropped by a faulty router on the way back to you.

so here we cannot know which exactly might have caused the timeout. so every state mutating operations must be idempotent.

3. There is no "now": according to Einstein's theory of relative states there is no absolute simultaneity. the same applies for distributed systems where we cant get the exact global current state at the moment.
   To handle this modern systems (flink) use markers, hen reached the marker it gets the state of each node like a snapshot and pass it to next node resulting in global snapshot even if they are happening at different times.

4. The Message-Passing Graph: Because there is no global clock, we dont measure time in seconds, but in causality.

You draw the nodes as vertical lines. You draw the messages as diagonal arrows. If your distributed transaction protocol requires Node 1 to know about Node 3's lock release, there must be an unbroken chain of arrows from Node 3 back to Node 1. If there isn't, your architecture has a race condition.

5. Asynchrony as an Adversary: do not design for happy path. build an adversary system to challenge the system as it tests realtime issues and how the system is currectly handling it.

6. End-to-End Argument: a common mistake is trusting the infrastructure to guarantee application correctness. 

ex: If your application crashes nanoseconds after the OS ACKs the TCP packet, that data is gone forever, even though the network layer says "delivered successfully." This is why robust pipelines don't just trust TCP; they use application-level offsets, logical checksums, and cryptographic hashing at the very edges of the system to verify the data was actually processed. The network in the middle is just a fast, unreliable pipe.

the atomic events in distributed system

1. internal computation

2. send data over network

3. receive data over network

timing models:

1. Synchronous: where we know after a certain time if node is not responding its 100% mathematically certainly dead.

2. Asynchronous: we dont know how long it will take to reply maybe 10s or 10 years.the CPU has to wait. here the timeout tell nothing because the other node might be dead or slow.

3. Partially Synchronous: the system behaves Asynchronous most of time, however network stablizes over time and latency and delay will become normal bound. this is called GST(global stabilization time).
